Overview of Geotoolkit.org (abridged Geotk).

Requirements

Geotk requires Java 6 Standard Edition. There is no additional requirement for metadata and referencing modules. But the coverage module requires the Java Advanced Imaging extension (see Running Geotk for download information). While not mandatory, users are encouraged to declare JavaDB in their classpath in order to take advantage of the embedded EPSG database. JavaDB is bundled in the Oracle distribution of JDK.

System initialization

Geotk does not require explicit initialization and shutdown for most cases. However a few services behave better when managed in an initialization - shutdown cycle. The {@link org.geotoolkit.lang.Setup} class provides a central place for initialization and shutdown, but developers can also gain more control by invoking themselves the various methods identified by the {@link org.geotoolkit.lang.Configuration} annotation.

Javadoc annotations

  • Versions

    The {@code @version} javadoc tag gives the last Geotk version when the code of this class got a significant change. For example if a class is annotated with {@code @version 3.00}, then this class got no significant change since the release of Geotoolkit.org 3.00 in June 2009.

    The {@code @since} javadoc tag gives the Geotk version when the annotated element (class, method, etc.) was first introduced. If the declared version is the same than the class {@code @version}, then that element is under development and uncommitted: it may change in any future snapshot and it will become stable only after the next release.

    Stable API (for example those annotated as {@code @since 2.4}) will not change lightly. If a stable API need to be removed, it is first annotated as {@code @deprecated} and will be removed only in a future release.

  • Authors

    The code were contributed by authors from different organisations. The organisation name appears between parenthesis after the author name, sometime in abbreviated form. More than one organisation may be listed for the same author if he continued to work on the code while moving to an other organisation. The organisation names used in this documentation are:

    Geomatys  Geomatys.
    Geosolutions  Geosolutions.
    HSRS  Help Service Remote Sensing.
    IRD  Institut de Recherche pour le Développement.
    MPO  Ministère des Pêches et Océans Canada.
    OpenGeo  OpenGeo.
    Refractions  Refractions Research.
    TOPP  The Open Planning Project.
    UBC  University of British Columbia.
    UWB  University of West Bohemia.
    USGS  U.S. Geological Survey. Actually not a direct contributor to Geotk, but their C/C++ code in the Proj4 project were used as a basis for the map projections.

Policies

  • Classes naming

    Implementations of GeoAPI interfaces usually (but not always) begin with {@code Default} or {@code Abstract} prefix. The later is used when a class is abstract according ISO specification; the class is not necessarily abstract in the Java sense. For example {@link org.geotoolkit.referencing.cs.AbstractCS} is abstract according ISO (users should always specify a more accurate subclasses), but can still be instantiated in Java. Actually it is instantiated in some exceptional cases (especially during WKT parsing) when the actual coordinate system type is really unknown.

    The {@code Default} or {@code Abstract} prefix cover the majority of cases for GeoAPI implementations, but there is a few exceptions. Implementations working only in space having a fixed number of dimensions are suffixed with {@code 1D}, {@code 2D}, {@code 3D} or {@code 4D}. The general implementation - if any - is prefixed with {@code General}. Example: {@link org.geotoolkit.geometry.Envelope2D} and {@link org.geotoolkit.geometry.GeneralEnvelope}.

    Classes that are not GeoAPI implementation are usually not prefixed by any of the above.

  • Logging

    Geotk uses the {@linkplain java.util.logging Java logging} by default, but can redirect its logging to Log4J or Apache's common-loggins. See {@link org.geotoolkit.util.logging} for details.
